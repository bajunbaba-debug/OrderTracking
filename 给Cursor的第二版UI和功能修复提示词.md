## 总体原则

1. 保持中文业务系统风格，页面要像可日常使用的订单/项目管理工具。
2. 不要新增营销页、欢迎页或大面积装饰。
3. 不要删除原始 Excel、数据库备份、PRD 文档。
4. 修改后请运行：
   - `npm run build`
   - 如有测试脚本，也运行测试脚本
5. 所有新增/编辑/标记完成/数据质量修改，必须真正写入数据库，并刷新页面数据。
6. 不要只改前端显示；涉及统计、筛选、字典、风险、质量问题时，请同步检查 API 和服务层逻辑。

## 全局 UI 问题

### 表格字符上下不对齐

当前多个表格中，文本、数字、标签、按钮上下视觉不齐。请统一所有业务表格样式：

1. 表头和单元格统一垂直居中。
2. 普通文本、数字、状态标签、操作按钮在同一行内居中对齐。
3. 多行字段要有明确规则，不要撑乱整行高度。
4. 数字类字段右对齐或等宽数字显示，至少要保证小数位视觉整齐。
5. 可在 `web/src/components/ui.tsx` 或全局 CSS 中沉淀统一 table class，避免每个页面各写一套。

重点检查：

3. 编辑页是否正确处理 loading、not found、错误状态。

### 4. 项目名称列固定两行

当前“项目名称”列因为字符串长度不同导致行高大小不一。

### 5. 风险列文案解释

当前“风险”列出现“算法关注”，普通用户看不懂。

请改为更容易理解的名称，并保留含义。建议：

1. “算法关注”改为“预计异常”或“计算异常提醒”。
2. 悬浮说明：表示该项目的单位工作量或预计工作日明显高于参考阈值，需要人工确认。
3. 风险清单页、明细页、统计标签中保持同一文案。

### 6. 标记完成要确认并记录日期

当前点击“标记完成”后，用户在 UI 上看不到是否记录了“设计完成”。

请改为：

1. 点击“标记完成”后先弹出确认框/弹窗，不要直接完成。
2. 弹窗中显示“设计完成日期”，默认当前日期。
3. 日期可以修改。
4. 用户确认后，把日期写入该项目的“设计完成”字段。
5. 保存后该项目状态变为已完成，并从默认“未完成”列表中消失。
6. 如果用户切到“已完成”或“全部”，应能看到该项目的设计完成日期。
7. 标记完成接口或服务层要明确接收并保存完成日期。

### 7. 新增项目明细中的字典统一管理

“新增项目明细”表单中的“类型”“类型细化”“常用备注”需要提供统一管理入口。

请实现：

1. 在新增/编辑表单中，靠近这些字段的位置提供“管理字典”或“管理选项”按钮。
2. 点击后可以管理：
   - 类型
   - 类型细化
   - 常用备注
3. “类型细化”必须隶属于某一个“类型”，每个“类型”对应自己独有的“类型细化”列表。
4. 当用户选择某个“类型”后，“类型细化”下拉框只显示该类型对应的细化项。
5. 如果修改了字典，新增/编辑表单中的选项要能刷新。
6. 现有从 Excel 导入的 Sheet2 字典数据要尽量保留并兼容。
